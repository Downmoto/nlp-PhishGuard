== Details of the Technical Solution Implemented

=== System Overview

PhishGuard is implemented as a modular Python project built around Hugging Face Transformers, PyTorch, pandas, and scikit-learn. The repository separates the pipeline into dedicated modules for data loading, text preprocessing, dataset preparation, training, evaluation, inference, and web serving. This structure allowed the project to remain testable and maintainable while still being simple to run through the scripts under `src/scripts/`.

At a high level, the implemented workflow is:

. Load labeled email data from the configured source file.
. Clean and normalize the text.
. Split the dataset into train, validation, and test partitions using stratification.
. Tokenize the text for BERT.
. Fine-tune the classifier with early stopping and checkpoint management.
. Evaluate the best checkpoint on the held-out test set.
. Serve predictions through a CLI tool and a Gradio web interface.

=== Preprocessing Pipeline

Raw email text is noisy. It may contain HTML fragments, URLs, repeated whitespace, and personally identifiable information such as phone numbers or email addresses. If that noise is left untouched, the model may overfit to incidental tokens rather than genuine phishing cues.

The implemented preprocessing pipeline applies four operations in a fixed order:

. *HTML stripping* using BeautifulSoup, with a regex fallback if BeautifulSoup is unavailable.
. *URL normalization* by replacing links with the placeholder token `[URL]`.
. *PII masking* by replacing email addresses with `[EMAIL]` and phone numbers with `[PHONE]`.
. *Whitespace normalization* to collapse repeated spaces and line breaks.

This design keeps the semantic structure of the email while reducing unnecessary variation. For example, two phishing emails containing different domains still share the same normalized `[URL]` token, allowing the model to focus on the surrounding language rather than memorizing a particular string.

=== Model Architecture

The core classifier is based on `bert-base-uncased`, a bidirectional transformer pre-trained on large general-domain corpora. The project wraps it with Hugging Face `BertForSequenceClassification` configured for two output labels:

. `0` for legitimate email
. `1` for phishing email

The maximum sequence length is set to 256 tokens. This is a practical tradeoff between retaining enough content from typical email bodies and keeping GPU memory usage under control. The final classification layer maps the contextualized BERT representation to a binary decision.

One implementation detail that improved efficiency is the custom `PhishingDataset` class. Instead of tokenizing every sample repeatedly on each epoch, the dataset pre-tokenizes the text once during initialization. That reduces CPU overhead during training and makes better use of the GPU.

The training loop also uses early stopping with a patience of 2 validation checks. That prevents unnecessary over-training and keeps the selected checkpoint aligned with generalization performance rather than just continued loss minimization on the training split.

Another practical feature is automatic checkpoint recovery. If training is interrupted, the trainer scans the output directory for the latest `checkpoint-*` folder and resumes from that point. This matters when working with long-running GPU experiments.

=== Evaluation Method

The final model is evaluated on the held-out test set, which was never used for optimization. The evaluation module computes:

. Accuracy
. Macro F1 score
. ROC-AUC
. Full classification report
. Confusion matrix
. ROC curve plot

These metrics were chosen because phishing detection has asymmetric risks. Accuracy is useful at a glance, but macro F1 gives a more balanced picture of per-class quality, and ROC-AUC measures ranking quality across thresholds.

=== Inference and Explainability

The inference pipeline is implemented in `PhishGuardPredictor`. For a single input email, it:

. Applies the same preprocessing pipeline used during training.
. Tokenizes the text using the saved tokenizer.
. Runs the fine-tuned classifier.
. Converts logits to probabilities through softmax.
. Returns the predicted label and confidence.
. Extracts token-level attention scores from the last transformer layer.

The explainability mechanism is intentionally lightweight. It uses the attention from the `[CLS]` token in the final layer, averaged across heads, to estimate which tokens received the most model focus. This does not provide a perfect causal explanation, but it gives the user a practical indication of the parts of the message that influenced the prediction.

=== User Interface and Deployment Path

To make the model accessible beyond offline experiments, the project includes both a command-line predictor and a Gradio web interface. The Gradio application accepts raw email text and returns three outputs:

. The predicted verdict
. The confidence score
. A short list of highly attended tokens

This interface is useful for demonstration, manual validation, and rapid usability testing. It also shows that the project is not only a model training exercise but a deployable NLP workflow with an end-user entry point.
